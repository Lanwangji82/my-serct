# QuantX 量化策略函数手册

## 1. 目标

本文档说明 QuantX 平台当前支持的 Python 策略编写方式，适用于研究、回测、纸面执行和后续沙盒执行。

当前平台策略主入口是一个固定函数：

```python
def generate_signals(frame: pl.DataFrame, params: dict) -> dict:
    ...
```

平台会把你的源码保存、编译检查，并在回测时直接调用这个函数。

## 2. 函数签名

必须实现：

```python
def generate_signals(frame: pl.DataFrame, params: dict) -> dict:
```

参数说明：

- `frame`
  - 类型：`polars.DataFrame`
  - 含义：平台传入的历史 K 线数据
  - 固定列：
    - `timestamp`
    - `open`
    - `high`
    - `low`
    - `close`
    - `volume`

- `params`
  - 类型：`dict`
  - 含义：策略参数
  - 来源：策略编辑器中的参数配置

返回值说明：

```python
{
    "entries": [...],
    "exits": [...],
}
```

- `entries`
  - 布尔序列
  - 长度必须与 `frame` 行数一致
  - `True` 表示该根 bar 产生开仓信号

- `exits`
  - 布尔序列
  - 长度必须与 `frame` 行数一致
  - `True` 表示该根 bar 产生平仓信号

支持返回类型：

- `list[bool]`
- `polars.Series`
- `pandas.Series`

## 3. 最小可运行示例

```python
import polars as pl


def generate_signals(frame: pl.DataFrame, params: dict) -> dict:
    fast_period = int(params.get("fastPeriod", 20))
    slow_period = int(params.get("slowPeriod", 50))

    if frame.height == 0:
        return {"entries": [], "exits": []}

    close = frame["close"]
    fast = close.rolling_mean(fast_period)
    slow = close.rolling_mean(slow_period)

    entries = []
    exits = []
    previous_fast = None
    previous_slow = None

    for current_fast, current_slow in zip(fast.to_list(), slow.to_list()):
        can_compare = None not in (previous_fast, previous_slow, current_fast, current_slow)
        entries.append(bool(can_compare and current_fast > current_slow and previous_fast <= previous_slow))
        exits.append(bool(can_compare and current_fast < current_slow and previous_fast >= previous_slow))
        previous_fast = current_fast
        previous_slow = current_slow

    return {"entries": entries, "exits": exits}
```

## 4. 平台当前内置运行环境

回测执行阶段当前保证可用的内容：

- `polars as pl`
- `math`
- Python 基础内置函数：
  - `abs`
  - `all`
  - `any`
  - `bool`
  - `dict`
  - `enumerate`
  - `float`
  - `int`
  - `len`
  - `list`
  - `max`
  - `min`
  - `range`
  - `round`
  - `sum`
  - `zip`

建议：

- 优先使用 `polars` 处理数据
- 不要在策略函数里做网络请求
- 不要在策略函数里直接下单
- 不要依赖本地文件读写

## 5. 设计约束

为了保证团队可维护和平台可审计，策略函数应遵循以下约束：

- 纯函数化
  - 相同 `frame + params` 应尽量返回相同结果

- 不做副作用
  - 不在函数里写数据库、发请求、修改系统状态

- 输出长度固定
  - `entries` 和 `exits` 必须与 `frame.height` 一致

- 参数外置
  - 周期、阈值、仓位大小尽量从 `params` 读取

## 6. 常见错误

### 6.1 缺少函数

错误：

```python
缺少 generate_signals(frame, params) 函数
```

原因：

- 没定义函数
- 函数名写错

### 6.2 返回长度不一致

错误示例：

- `entries` 数量和 K 线数量不一致
- `exits` 数量和 K 线数量不一致

原因：

- 只返回了局部区间信号
- 没有为前几根指标未就绪的 K 线补 `False`

### 6.3 语法错误

平台会在“编译检查”阶段直接返回行号和错误信息。

## 7. 推荐开发流程

1. 在平台新建 Python 策略
2. 填写策略名称、交易对、周期、参数、风险限制
3. 编写 `generate_signals(frame, params)`
4. 点击“编译检查”
5. 保存到平台
6. 执行回测
7. 检查收益、回撤、交易次数
8. 再决定是否进入纸面或沙盒执行

## 8. 当前建议的函数边界

建议把策略逻辑限制在以下内容：

- 指标计算
- 信号生成
- 开平仓规则判断

不建议直接塞进策略函数的内容：

- 仓位风控审批
- 多账户路由
- API 凭证处理
- Broker 连接管理

这些应该由平台的执行与治理层处理。
