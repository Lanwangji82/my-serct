# QuantX 平台策略落盘与治理说明

## 1. 为什么要落盘

企业级量化平台不能只把策略存在数据库里。策略需要同时满足：

- 可追溯
- 可审计
- 可回滚
- 可交接
- 可被研发和风控共同查看

因此，QuantX 当前会在“保存策略”时，把 Python 策略源码和元数据都写入文件系统。

## 2. 当前默认落盘目录

默认目录：

```text
python_services/strategy_store/
```

可通过环境变量修改：

```env
PY_PLATFORM_STRATEGY_STORE=D:/quantx-platform/python_services/strategy_store
```

## 3. 目录结构

每个策略一个独立目录，目录命名规则：

```text
{策略名slug}__{strategy_id}
```

例如：

```text
python_services/strategy_store/
  BTC-SMA-趋势__strat_abcd1234/
    latest.py
    latest.json
    v1742659200000.py
    v1742659200000.json
```

## 4. 文件含义

### `latest.py`

- 当前最新版本策略源码
- 适合平台直接展示和人工查阅

### `latest.json`

- 当前最新版本元数据
- 包含：
  - 策略 id
  - 名称
  - 描述
  - 交易对
  - 周期
  - 市场类型
  - 运行环境
  - 参数
  - 风险限制
  - 编译结果
  - 版本号

### `v{timestamp}.py`

- 历史版本源码
- 每次保存产生一个新版本

### `v{timestamp}.json`

- 对应版本的元数据快照

## 5. 平台界面如何显示

当前策略工作流中，策略保存成功后会展示：

- 策略目录
- 最新源码文件
- 最新元数据文件
- 版本号

这意味着平台用户可以明确知道：

- 每个策略保存到了哪里
- 当前运行的是哪个版本
- 如何把策略文件交给同事复查

## 6. 适合团队治理的使用方式

建议团队按以下方式使用：

1. 研究员在平台内编写并保存策略
2. 平台生成源码文件与元数据快照
3. 风控/负责人审核 `latest.json` 中的参数和限制
4. 审核通过后再进入纸面或沙盒执行

## 7. 建议后续增强

当前已具备“平台保存 + 文件落盘”能力，下一阶段建议继续补：

- Git 同步
  - 将落盘目录同步到仓库或内部代码库

- 发布审批流
  - 研究通过后才能进入沙盒
  - 沙盒通过后才能进入生产

- 策略标签体系
  - 团队
  - 作者
  - 风险等级
  - 适用市场

- 版本对比
  - 直接对比 `latest.py` 与历史版本差异

- 审计签名
  - 记录谁保存、谁审核、谁发布

## 8. 结论

当前平台中，Python 策略已经不是“前端临时文本”，而是正式落盘的研究资产。  
这一步是从个人工具走向公司平台的关键基础。
