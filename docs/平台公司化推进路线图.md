# QuantX 平台公司化推进路线图

## 1. 当前已经具备的基础

平台当前已经具备：

- Python 策略编辑
- 编译检查
- 策略保存
- 策略文件落盘
- Polars 数据处理
- VectorBT 回测
- CCXT 执行适配
- OKX Demo 模拟交易验证
- 凭证存储
- 基础审计事件

这说明平台已经从“原型界面”进入“可持续工程化”的阶段。

## 2. 下一阶段应优先建设的模块

### 2.1 策略资产管理

- 策略标签
- 作者/团队字段
- 策略状态
  - 草稿
  - 待审核
  - 已批准
  - 已禁用

### 2.2 发布治理

- 回测通过门槛
- 风险审查
- 沙盒审批
- 生产发布审批

### 2.3 研究可复现

- 固定数据快照
- 固定参数快照
- 固定回测版本
- 记录依赖版本

### 2.4 执行可观测

- 每次请求耗时
- 每次下单结果
- 每次拒单原因
- 代理/网络状态快照

### 2.5 组合与风险

- 组合级别限额
- 策略级别限额
- 交易对级别限额
- 每日损失限制
- 熔断与停机机制

## 3. 推荐组织方式

建议把 QuantX 拆成四类角色：

- 研究
  - 写策略
  - 回测
  - 参数实验

- 风控
  - 审核风险参数
  - 审核允许标的
  - 审核最大杠杆和损失限制

- 执行
  - 管理 broker
  - 管理连接与代理
  - 管理沙盒/生产环境

- 治理
  - 凭证与审计
  - 发布流程
  - 权限控制

## 4. 建议的落地顺序

### 第一阶段

- 策略落盘
- 回测稳定
- 策略文档
- 凭证治理

### 第二阶段

- 策略审批流
- 版本对比
- 回测基线报告
- 执行联通性检查

### 第三阶段

- 多 broker 执行
- 组合级风控
- 告警系统
- 任务调度系统

### 第四阶段

- 团队权限模型
- Git / CI 集成
- 生产发布门禁
- 统一审计报表

## 5. 当前最值得继续做的三件事

1. 给策略增加“草稿 / 审核中 / 已批准 / 已停用”状态机
2. 给策略落盘目录接 Git 版本管理
3. 给执行模块增加正式的“发布前检查”清单

## 6. 结论

平台已经有了可以继续工业化推进的基础。  
后续不要再按“修页面”的思路做，而要按“策略资产、执行资产、治理资产”三条线持续建设。
